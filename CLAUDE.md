# CLAUDE.md

This file provides guidance to Claude Code (`claude.ai/code`) when working with code in this repository.

## Project Overview

This repository is a static GitHub Pages site about German elections and polling. It now has three user-facing pages under `docs/`:

- `docs/index.html` — federal polling trends from Forschungsgruppe Wahlen / Politbarometer
- `docs/demographics.html` — historical Bundestag demographic voting patterns from Bundeswahlleiterin Heft 4
- `docs/state-polling.html` — Bundesland left-vs-right polling map and table from `dawum.de`

There is no frontend build step and no npm toolchain. The site is plain HTML/CSS/ES modules, with committed JSON data files under `docs/data/`.

## Commands

Set up the local Python environment:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Refresh federal polling data:

```bash
python3 scripts/fetch_data.py
```

Refresh state polling data:

```bash
python3 scripts/fetch_state_data.py
```

Re-extract demographics data from the Bundeswahlleiterin CSV:

```bash
python3 scripts/extract_demographics.py
```

Refresh state population data (manual; needs a GENESIS-Online API token in `.env`):

```bash
python3 scripts/fetch_population.py
```

Run a local dev server:

```bash
python3 -m http.server -d docs
```

There are currently no tests, linters, or build commands.

## Deployment And Automation

- The site is structured for GitHub Pages from `docs/`.
- `.github/workflows/update-data.yml` runs daily at `06:00 UTC` and also supports manual dispatch.
- It has a `concurrency` group (`update-data`, `cancel-in-progress: false`) so a manual dispatch and the cron never race, a `timeout-minutes: 15` ceiling, a `git pull --rebase origin main` before pushing, and an `if: failure()` step that emits a `::error` annotation.
- The workflow uses Python `3.12`, installs from the pinned `requirements.txt` (`pip install -r requirements.txt`), then runs:
  - `python scripts/fetch_data.py`
  - `python scripts/fetch_state_data.py`
- If data changed, it commits:
  - `docs/data/polling.json`
  - `docs/data/state-polling.json`
  - `docs/data/state-polling.last_update.txt`
- `docs/data/state-polling.last_update.txt` is committed deliberately: it is the state pipeline's lightweight change detector, and committing it lets the detector survive ephemeral runners without `actions/cache`.
- There is **no** committed `.etag` file. The federal ETag cache was dropped (its `<mtime>-<hash>` value churned on every fetch and produced spurious daily commits); `scripts/fetch_data.py` now short-circuits by comparing freshly parsed records against the committed `polling.json`.

`docs/data/pre-polling.json` is committed to the repo and maintained separately from the daily GitHub Action. `scripts/fetch_data.py` reads it and prepends those rows when writing `docs/data/polling.json`.

The demographics JSON and the population JSON are committed to the repo but are **not** regenerated by the GitHub Action — `scripts/extract_demographics.py` and `scripts/fetch_population.py` are manual, low-frequency scripts (demographics changes once per federal election; population once per year).

## Data Pipelines

### 1. Federal polling (`scripts/fetch_data.py`)

- Downloads the Excel workbook from Forschungsgruppe Wahlen:
  - `https://www.forschungsgruppe.de/.../1_Projektion_1.xlsx`
- Discovers the workbook link from `INDEX_URL` (the filename churns upstream) and pins the resolved/redirected host to `forschungsgruppe.de` before trusting the bytes.
- No ETag cache: it always downloads, then skips the write when the freshly parsed records equal the committed `docs/data/polling.json` (no `.etag` file, no `304` branch).
- Locates the header row by scanning each sheet for one containing both `CDU/CSU` and `SPD`, then matches party columns by header label (tolerant of sheet renames, row shifts, and reordered columns); the leftmost remaining non-empty column is the date.
- Maps Excel headers to the JSON keys:
  - `cdu`, `spd`, `gruene`, `fdp`, `linke`, `piraten`, `afd`, `fw`, `bsw`
- Coerces party cells to numeric, warns to stderr on present-but-blank cells (within a party's active span) and on rows summing `>100`, then computes `sonstige` as `100 - sum(other parties)` clipped at `0`.
- Prepends committed historical rows from `docs/data/pre-polling.json`, filtering out rows that overlap the fetched Excel series (and logging how many fetched rows were dropped).
- Routes the write through a non-empty / no-shrink guard, and writes `docs/data/polling.json` with top-level shape:
  - `updated`
  - `source`, `license`
  - `data: [{ date, cdu, spd, ... }]`
- Optional opt-in cross-check: with `FGW_CROSSCHECK=1`, `crosscheck_against_dawum()` warns (read-only, never a source) if the latest FGW row disagrees with dawum's FGW Bundestag row beyond tolerance.

Important implementation detail: the script uses `StrEnum`, so local runs expect Python `3.11+`.

### 2. State polling (`scripts/fetch_state_data.py`)

- Fetches:
  - `https://api.dawum.de/newest_surveys.json`
  - `https://api.dawum.de/last_update.txt`
- Uses `docs/data/state-polling.last_update.txt` as a lightweight change detector.
- Computes a weighted average per Bundesland across recent state surveys:
  - per-state survey weight = `sqrt(sample_size) * 0.5^(age_days/30)`
- The top-level **summary** stats (`avg_diff`, `west_avg`, `east_avg`) are weighted differently — by each state's population from `docs/data/population.json` (falling back to a weight of `1.0` per state when that file is absent). This is distinct from the per-state survey weighting above.
- Defines:
  - left = `SPD + Grüne + Linke`
  - right = `CDU/CSU + AfD`
- Falls back to `scripts/state_election_results.json` if a state has no current surveys; each state object carries an `is_fallback` boolean marking whether it came from that stale-election fallback vs. live surveys.
- Asserts exactly 16 states and routes the write through the non-empty / no-shrink guard.
- Writes `docs/data/state-polling.json` with:
  - `updated`
  - `source`
  - `summary`
  - `states` (each with `is_fallback`)

The state page also depends on the committed map asset `docs/data/germany-states.svg`.

### 3. Demographics (`scripts/extract_demographics.py`)

- One-off/manual extraction script (NOT on the daily Action). It no longer uses `pdfplumber`/the PDF.
- Discovers and downloads the official labeled CSV (`btw_rws_zwst-1953.csv`) from the Bundeswahlleiterin "Repräsentative Wahlstatistik" page.
- Parses it with `csv.reader(..., delimiter=";")`, keyed on header **labels** (`Bundestagswahl`, `Geschlecht`, `Altersgruppe`, and party columns) rather than positional columns — so an upstream reflow can't silently misalign parties. Übersicht 9: "Zweitstimmen nach Geschlecht und Altersgruppen seit 1953".
- German decimals (`29,9`) → floats; empty/`–`/`.`/`...` cells → `null` (not `0`).
- Handles:
  - normalized age-bracket keys
  - gender buckets `insgesamt`, `frauen`, `maenner`
  - CSU embedded in CDU for `1953` and `1957`
- Asserts header labels, year/gender coverage, then writes `docs/data/demographics.json` (top-level `source`, `license`, `parties`, `genders`) through the non-empty / no-shrink guard.

Treat `docs/data/demographics.json` as the frontend source of truth. Any regeneration must reproduce it 1:1; the only permitted diff is additive metadata (`source`/`license`) — never hand-edit values.

### 4. Population (`scripts/fetch_population.py`)

- **Manual** script — NOT run by the daily Action.
- Fetches official state population (Destatis table `12411-0010`) via the `pystatis` library, which needs a GENESIS-Online API token in `.env` as `PYSTATIS_GENESIS_API_USERNAME`.
- Writes `docs/data/population.json` with top-level `updated`, `source`, `license`, and `data` (`{ "DE-XX": int }` per state) through the non-empty / no-shrink guard.
- Drives the population-weighted summary stats on the state-polling page (see the State polling section above).
- Currently stale at `2024-12-31`; refresh annually.

## Frontend Architecture

### Shared

- `docs/css/style.css` contains the shared styling for all three pages.
- `docs/js/config.js` is the shared federal-party config:
  - labels
  - colors
  - block membership
  - election marker dates
  - EWMA halflife (`30` days)
- `docs/fonts/` and `docs/js/vendor/plotly.min.js` are self-hosted.

### Federal polling page

Entry point and modules:

- `docs/index.html`
- `docs/js/main.js`
- `docs/js/data.js`
- `docs/js/charts.js`
- `docs/js/config.js`

Behavior:

- Loads `docs/data/polling.json`
- Converts date strings to `Date`
- Computes block totals dynamically from `PARTY_CONFIG.block`
- Renders two Plotly charts:
  - major parties
  - political blocks
- Applies EWMA smoothing with a 30-day halflife
- Shows election markers from `ELECTION_DATES`
- Uses mobile/desktop layout differences at `768px`

`MAJOR_PARTIES` in `docs/js/config.js` controls which parties appear in the party chart.

### Demographics page

Entry point and modules:

- `docs/demographics.html`
- `docs/js/demographics-main.js`
- `docs/js/demographics-data.js`
- `docs/js/demographics-charts.js`
- `docs/js/demographics-config.js`

Behavior:

- Loads `docs/data/demographics.json`
- Lets the user build up to `5` comparison lines
- Each line combines:
  - gender
  - age bracket
  - one or more parties
- `union` is a virtual frontend-only party combining `CDU + CSU`
- Missing age buckets create chart gaps; missing party values within an existing row are treated as `0`
- Renders a single Plotly line chart with configurable selections

### State polling page

Entry point and modules:

- `docs/state-polling.html`
- `docs/js/state-polling-main.js`
- `docs/js/state-polling-data.js`
- `docs/js/state-polling-charts.js`
- `docs/js/state-polling-config.js`

Behavior:

- Loads `docs/data/state-polling.json`
- Fetches and inlines `docs/data/germany-states.svg`
- Renders:
  - a summary segment bar
  - a current-diff map
  - a change-vs-election map
  - a sortable table
- Uses direct DOM/SVG manipulation rather than Plotly for the maps
- Colors are derived from a diverging scale centered at `0`, with config in `state-polling-config.js`

## Dependencies

`requirements.txt` is fully pinned (`==`) and contains:

- `httpx` — used by the daily Action (federal + state fetch)
- `pandas` — used by the daily Action (federal Excel parsing)
- `openpyxl` — used by the daily Action (Excel engine for pandas)
- `pystatis` — **manual-only**, used solely by `scripts/fetch_population.py`; the daily Action installs it but never invokes it

(`pdfplumber` was removed when the demographics extractor migrated from the PDF to the labeled CSV.)

Frontend dependencies are vendored or self-hosted; there is no package manager lockfile or JS build process.
